## Instructions

1. Create a branch named `lean-squad-ci/{{.Vessel.ID}}`.
2. Commit only `.github/workflows/lean-ci.yml` with a Conventional Commits
   message such as `ci(lean-squad): add Lean 4 build workflow`.
3. Push the branch.
4. Open the PR with:

   ```
   gh pr create \
     --title "[Lean Squad] Add Lean 4 CI" \
     --label "ready-to-merge" \
     --body "<body>"
   ```

The PR body must include:

- A one-sentence statement that this PR was generated by the xylem
  `lean-squad-ci` workflow, with a 🔬 disclosure marker.
- A link to the upstream doctrine:
  `https://github.com/githubnext/agentics/blob/main/docs/lean-squad.md`.
- A short explainer for maintainers without Lean 4 background:
  "Mathlib is the Lean 4 math library. A from-scratch Mathlib build costs
  ~30 minutes of CI time. `lake exe cache get` downloads the Mathlib
  community cache of prebuilt `.olean` files, which cuts this job to ~2 min.
  The trigger is path-scoped to `formal-verification/lean/**` so this
  workflow does not run on PRs that do not touch Lean files."
- A trailer line marker for the `lean-squad` tick coordinator to ingest:

  ```
  LEAN_SQUAD_RUN: {"task":"ci","target":"lean-ci","status":"ok","artefact":".github/workflows/lean-ci.yml"}
  ```

Do NOT modify any file other than `.github/workflows/lean-ci.yml` in this
PR. If the only diff is the run marker in the PR body, that's fine — the
workflow file was already in the desired state and the prior phase should
have emitted `XYLEM_NOOP`. If you reach this phase with no file changes to
commit, close out with "PR: none — lean-ci.yml already satisfies the spec"
and do not open a PR.
