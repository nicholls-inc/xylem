## Write the Lean file

Write `formal-verification/lean/FVSquad/<Target>.lean` (the exact PascalCase `<Target>` you derived in the analysis phase).

### File structure — in this order

1. **Header comment** with the 🔬 disclosure and vessel ID:
   ```lean
   -- 🔬 Lean Squad formal spec — generated by xylem `lean-squad-formal-spec` workflow
   -- Vessel: {{.Vessel.ID}}  Ref: {{.Vessel.Ref}}
   -- Bodies are `sorry` pending Task 4 (lean-squad-extract-impl) and Task 5 (lean-squad-prove).
   ```
2. **Imports** — only those you actually need. `import Mathlib.Data.List.Basic`, `import Mathlib.Tactic`, etc.
3. **Namespace** — `namespace FVSquad.<Target>` / `end FVSquad.<Target>` at the bottom.
4. **Type definitions** — `structure`/`inductive` for any domain types the spec describes.
5. **Function signatures** — one `def` per operation, body `:= sorry`.
6. **Theorem declarations** — one `theorem` per universal claim in the spec, body `:= by sorry` (or `:= sorry`).

### Teach Lean syntax inline

This workflow's core promise is "zero FV expertise". On the first use of any Lean construct, add a short line comment explaining it. Examples:

```lean
-- `structure` declares a record type with named fields.
structure Tree (α : Type) where
  root : α
  children : List (Tree α)

-- `def f : α → β := sorry` declares a function from α to β whose body we will fill in later.
-- `sorry` is a placeholder term that typechecks as any type — the file still compiles.
def height {α : Type} (t : Tree α) : Nat := sorry

-- `theorem` states a proposition; `by sorry` skips the proof while keeping the claim.
theorem height_nonneg {α : Type} (t : Tree α) : height t ≥ 0 := by sorry
```

Do this only on the *first* use of each construct — not every time. Think "a brief Lean-4 primer woven into the file" rather than "a textbook".

### Keep all bodies as `sorry`

Do **not** attempt to prove anything or fill in any function body. Even if the answer is obvious, write `sorry`. Task 4 (`lean-squad-extract-impl`) will fill in function bodies; Task 5 (`lean-squad-prove`) will discharge the theorems. Mixing those into this phase breaks the shared-file-writer contract.

### Make it compile

The only hard requirement on the file is: `lake build` succeeds. `sorry` always typechecks, so the usual failure modes are:
- wrong imports,
- a type name that doesn't exist,
- a typo in a constructor or field name.

Keep the file minimal — a handful of types, a handful of signatures, a handful of theorem statements. A 60-line file with a dozen `sorry`s is the expected output, not a 400-line file.
